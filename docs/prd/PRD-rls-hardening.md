# PRD: RLS Hardening — Close the Three White-Label Tenancy Leaks

- **Version:** 1.0
- **Date:** 2026-06-01
- **Status:** Draft / Ready for implementation
- **Owner:** Alex + clawdbot
- **Source of truth for gaps:** `/root/chess-app/docs/archive/RLS-FAILURES.md` (generated by `backend/tests/test_rls_isolation.py` against live Supabase `qtzujwiqzbgyhdgulvcd`)
- **Related ADRs:** ADR-0003 (Clerk auth), ADR-0005 (subdomain-per-tenant)

## 1. Why

The white-label PRD (`PRD-whitelabel-ratings-calendar.md`) shipped code for Phases 1–6 + 7A/7B and the migrations were applied today. The RLS fuzzer (`test_rls_isolation.py`) immediately surfaced three real holes that mean **we cannot onboard a second school** until they are closed:

1. **6 tables have no RLS at all** — `tournaments`, `tournament_registrations`, `tournament_games`, `tournament_standings`, `player_ratings`, `rating_history`. Anon/authenticated callers via PostgREST or the Supabase JS client can read every school's data.
2. **`organization_members` policy recurses into itself** — Postgres aborts with `infinite recursion`. Cascades to every dependent policy, so currently *no* authenticated read of any org-scoped table works via the Supabase client. The 4 xpassed tests in the suite are deny-by-accident.
3. **`auth.uid()` returns `uuid`, Clerk IDs are `text`** — every policy that calls `auth.uid()` crashes under a real production Clerk token (`invalid input syntax for type uuid`). The whole RLS scheme is non-functional in prod.

Today the Flask backend uses the service-role key and bypasses RLS, so users do not yet see broken queries. That hides the issue but does not fix it: RLS is our **defense-in-depth** layer for the future moment any client (frontend, mobile, external integration) reads Supabase directly with an anon/authenticated key. The white-label PRD assumes this layer exists.

## 2. Goals

- Fix all three gaps with the smallest surface change that holds.
- After fixes, **every xfail in `test_rls_isolation.py` should become xpass** (or be promoted to a regular pass).
- No regression to current Flask-mediated reads (service-role bypasses RLS — still works).
- No change to user-visible behaviour.
- Documented rollback path for every migration.

## 3. Non-goals

- Do **not** invent `coaching_sessions` or `user_chess_profiles` tables. The guarded blocks in migration 005 stay guarded.
- Do **not** migrate `rating_history` to carry `organization_id` — it currently has no org column. Scope it via its `tournament_id` join instead. (See §4.4.)
- Do **not** flip the Flask backend off the service-role key. That is a separate, larger change.
- Do **not** touch the existing seeded `demo` org (id `08653c5f-ac6b-4f63-83c4-edecf0f91207`) — it is the reference org for `demo.chesster.io` SSL verification.

## 4. Changes

### 4.1 New SQL migration: `20260601_008_rls_hardening.sql`

Single migration that does all of the following, in this order. Idempotent (uses `CREATE OR REPLACE`, `DROP POLICY IF EXISTS`, guards).

**A. `clerk_uid()` helper function**

```sql
CREATE OR REPLACE FUNCTION public.clerk_uid()
RETURNS text
LANGUAGE sql
STABLE
AS $$
  SELECT nullif(current_setting('request.jwt.claims', true), '')::jsonb ->> 'sub';
$$;

REVOKE ALL ON FUNCTION public.clerk_uid() FROM public;
GRANT EXECUTE ON FUNCTION public.clerk_uid() TO anon, authenticated, service_role;
```

This reads the JWT `sub` claim as `text`, never casts to `uuid`, and works for both Clerk IDs (`user_2abc...`) and UUID-shaped test IDs (fuzzer harness).

**B. `is_org_member(org uuid)` and `is_org_role(org uuid, roles text[])` SECURITY DEFINER helpers**

```sql
CREATE OR REPLACE FUNCTION public.is_org_member(org uuid)
RETURNS boolean
LANGUAGE sql
SECURITY DEFINER
STABLE
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.organization_members om
    WHERE om.organization_id = org
      AND om.user_id = public.clerk_uid()
  );
$$;

CREATE OR REPLACE FUNCTION public.is_org_role(org uuid, roles text[])
RETURNS boolean
LANGUAGE sql
SECURITY DEFINER
STABLE
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.organization_members om
    WHERE om.organization_id = org
      AND om.user_id = public.clerk_uid()
      AND om.role = ANY(roles)
  );
$$;

REVOKE ALL ON FUNCTION public.is_org_member(uuid) FROM public;
REVOKE ALL ON FUNCTION public.is_org_role(uuid, text[]) FROM public;
GRANT EXECUTE ON FUNCTION public.is_org_member(uuid) TO anon, authenticated, service_role;
GRANT EXECUTE ON FUNCTION public.is_org_role(uuid, text[]) TO anon, authenticated, service_role;
```

`SECURITY DEFINER` runs the lookup as the function owner (`postgres`), bypassing the recursive RLS check on `organization_members`. `SET search_path = public` prevents schema-hijack abuse.

**C. Rewrite every existing policy in migration 005 to use the helpers**

For each policy in `20260428_005_rls_policies.sql` that subqueries `organization_members`:

1. `DROP POLICY IF EXISTS "<name>" ON <table>;`
2. `CREATE POLICY "<name>" ON <table> ... USING (is_org_member(<table>.organization_id))` — or `is_org_role(...)` for admin/owner/teacher gated policies.
3. Replace every `auth.uid()::text` with `clerk_uid()`.

Apply to: `organizations`, `organization_members`, `organization_content`, `organization_billing`, `user_progress`, `lesson_chat_history`, `user_games`. Keep the same policy names so the fuzzer's positive/negative assertions still match. Keep the `coaching_sessions` / `user_chess_profiles` guard blocks in place (just swap the helper inside them).

**D. Enable RLS on the 6 tournament/rating tables and write policies**

```sql
ALTER TABLE tournaments ENABLE ROW LEVEL SECURITY;
ALTER TABLE tournament_registrations ENABLE ROW LEVEL SECURITY;
ALTER TABLE tournament_games ENABLE ROW LEVEL SECURITY;
ALTER TABLE tournament_standings ENABLE ROW LEVEL SECURITY;
ALTER TABLE player_ratings ENABLE ROW LEVEL SECURITY;
ALTER TABLE rating_history ENABLE ROW LEVEL SECURITY;
```

Policies:

- **`tournaments`** — has `organizer_org_id`. Public read for `status IN ('published','ongoing','completed')` (so the public calendar still works on each subdomain). Org-member read for any status. Org-admin/owner manage.
- **`tournament_registrations`** — scope via `tournaments` join: `is_org_member((SELECT organizer_org_id FROM tournaments WHERE id = tournament_registrations.tournament_id))`. Plus a self-row policy: the registered player can always see their own registration (`user_id = clerk_uid()`).
- **`tournament_games`** — public read for games belonging to a published/completed tournament. Org-member read for any. Org-admin manage.
- **`tournament_standings`** — same shape as `tournament_games`.
- **`player_ratings`** — has `organization_id`. Public read (ratings are intentionally visible on each school's leaderboard). Org-admin/owner/teacher manage.
- **`rating_history`** — no `organization_id` column. Scope via `player_ratings` join: `is_org_member((SELECT organization_id FROM player_ratings WHERE id = rating_history.player_rating_id))`. Plus a self-row policy: the player can see their own history (`user_id = clerk_uid()`).

All `USING` and `WITH CHECK` clauses must use the helpers (`is_org_member`, `is_org_role`) and `clerk_uid()` — never `auth.uid()` and never inline subqueries against `organization_members`.

**E. Rollback section**

Append a commented-out `-- ROLLBACK` block at the end of the migration with the exact `DROP POLICY` / `DROP FUNCTION` / `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` statements to undo every change, in reverse order. Comment-only so it does not run on apply, but copy-pasteable for emergency revert.

### 4.2 Update `test_rls_isolation.py`

- Remove `xfail` from every assertion that targeted gaps 1, 2, 3. They should now pass.
- Add **new positive tests** for the 6 newly-protected tables: cross-org read returns 0 rows, same-org read returns expected rows.
- Add a positive test that `clerk_uid()` returns the JWT `sub` as text for a non-UUID Clerk-shaped value (e.g. `user_2abcDEF`).
- Add a positive test that `is_org_member` does **not** trigger recursion (success = no `InvalidObjectDefinition` raised).
- Keep the existing safety guards: random-suffix slugs, mandatory cleanup, never touch the seeded `demo` org row.

### 4.3 No Clerk JWT template change

The PRD originally suggested either a Postgres helper or a Clerk JWT template tweak. Choosing the Postgres helper (`clerk_uid()`) keeps the change inside the DB layer — no Clerk dashboard surgery, no risk to the auth flow, no Vercel redeploy. ADR-0003 already documents Clerk as the trust anchor; this PRD does not change that.

### 4.4 No `rating_history.organization_id` backfill

Scoping via the `player_ratings` join keeps the schema normalised and avoids a migration that would need to backfill every existing row. Cost: one extra subquery per RLS check. Acceptable.

## 5. Apply procedure

1. `cd /root/chess-app`
2. Read `backend/.env` for `SUPABASE_DB_URL`.
3. Apply migration on the live DB:
   ```
   psql "$SUPABASE_DB_URL" -f supabase/migrations/20260601_008_rls_hardening.sql
   ```
4. Run the fuzzer:
   ```
   cd backend && \
     SUPABASE_DB_URL="$SUPABASE_DB_URL" \
     venv/bin/python -m pytest tests/test_rls_isolation.py -v
   ```
5. Expected: **all xfails flip to pass (or to xpass with strict=False then upgraded to pass)**. New positive tests pass. Zero failures.
6. Smoke-check the live app:
   - `curl https://chesster.io` → 200
   - `curl -I https://demo.chesster.io` → 200 or org-404 (still SSL-issuing is OK)
   - One logged-in path through `/admin/dashboard` and `/tournaments` (manual) — the service-role-keyed backend should be unaffected.

## 6. Risk and rollback

| Risk | Mitigation |
| --- | --- |
| Helper function `SECURITY DEFINER` runs as `postgres` and could become an attack surface if extended carelessly | Keep them tiny, `STABLE`, `SET search_path = public`, never accept user-controllable SQL fragments |
| New RLS on tournament tables breaks an unnoticed client-side direct read | Backend reads use service-role and bypass RLS; frontend uses backend; risk is low. Manual smoke check covers public tournament page |
| Wrong policy lets cross-org read slip through | Fuzzer is the gate. Don't merge unless all xfails flip green and new positive tests pass |
| Live psql apply errors mid-migration | Migration is idempotent (`CREATE OR REPLACE`, `DROP POLICY IF EXISTS`); re-run safely. Rollback block at bottom of migration |
| `clerk_uid()` returns null for unauthenticated calls | Existing `auth.uid()` does the same; policies already tolerate this |
