# RLS Isolation Failures

Generated by `backend/tests/test_rls_isolation.py` on **2026-06-01**, against
the live Supabase project `qtzujwiqzbgyhdgulvcd`. These are the cases where
the test suite found that the database **does not** enforce cross-org
isolation as the migrations intend.

## 1. RLS not enabled — tables have no policies at all

These tables hold per-org data but `ALTER TABLE … ENABLE ROW LEVEL SECURITY`
was never run on them. Any logged-in user (or the public `anon` role, via
PostgREST/Supabase JS) can read **every** row.

| Table | Has `organization_id`? | Has any RLS policy? |
| --- | --- | --- |
| `tournaments` | yes (`organizer_org_id`) | no |
| `tournament_registrations` | indirect (via tournament) | no |
| `tournament_games` | indirect (via tournament) | no |
| `tournament_standings` | indirect (via tournament) | no |
| `player_ratings` | yes | no |
| `rating_history` | **no column at all** | no |

Reproduction:
```sql
SELECT relname, relrowsecurity
FROM pg_class
WHERE relname IN ('tournaments', 'tournament_registrations',
                  'tournament_games', 'tournament_standings',
                  'player_ratings', 'rating_history');
-- relrowsecurity = false for all six
```

Migration `20260428_005_rls_policies.sql` covers
`organizations / organization_members / organization_content /
organization_billing / user_progress / lesson_chat_history / user_games`.
The tournament/rating tables introduced by migrations 003 and 004 were never
included.

Additionally, `rating_history` has no `organization_id` (or `organizer_org_id`)
column whatsoever — there is no scoping to enforce, even if RLS were enabled.

**Severity:** high. The tournament calendar, registrations, games, standings,
and per-org rating ladder all leak across tenants today.

## 2. `organization_members` policy recurses into itself

The `org_members_read` policy on `organization_members` is:

```sql
USING (
  EXISTS (
    SELECT 1 FROM organization_members om
    WHERE om.organization_id = organization_members.organization_id
    AND om.user_id = auth.uid()::text
  )
)
```

The subquery hits `organization_members`, which re-evaluates the same policy,
which hits `organization_members` again, … Postgres detects this and aborts:

```
psycopg2.errors.InvalidObjectDefinition:
  infinite recursion detected in policy for relation "organization_members"
```

This cascades to **every other RLS policy that subqueries `organization_members`**
(`organization_content`, `organization_billing`, `user_progress`,
`lesson_chat_history`, `user_games` — i.e. essentially all of them).
Authenticated reads of those tables under the `authenticated` role currently
raise this error rather than returning rows.

The standard fix is a `SECURITY DEFINER` helper function that bypasses RLS
while looking up membership:

```sql
CREATE FUNCTION public.is_org_member(org uuid)
RETURNS boolean LANGUAGE sql SECURITY DEFINER STABLE
AS $$
  SELECT EXISTS (
    SELECT 1 FROM organization_members om
    WHERE om.organization_id = org
      AND om.user_id = auth.uid()::text
  );
$$;
```

…and then call `is_org_member(organization_id)` from every policy instead of
inlining the EXISTS clause.

**Severity:** high. Right now no authenticated user can read any org-scoped
table at all via the standard Supabase client.

## 3. `auth.uid()` returns `uuid`, but `organization_members.user_id` is `text` Clerk IDs

`auth.uid()` is defined to cast the JWT `sub` claim to `uuid`:

```sql
(nullif(current_setting('request.jwt.claims', true), '')::jsonb ->> 'sub')::uuid
```

Real Clerk JWTs carry `sub` values like `user_2abcDEF...` — those are not
valid UUIDs and will crash `auth.uid()` outright (`invalid input syntax for
type uuid`), which means every Supabase-side policy that calls `auth.uid()`
will fail under a real Clerk token.

The test suite works around this by using UUID-shaped fake user IDs, so the
isolation logic can be exercised. In production, this needs either:

* a Postgres function override that reads `sub` as `text` (e.g.
  `public.clerk_uid()` returning text), and policies that use it; or
* configuring Clerk's JWT template to put the Clerk ID into a custom claim
  and ignoring `sub` in policies.

**Severity:** medium-high. Not a leak per se, but the entire RLS scheme is
non-functional with real production tokens until this is reconciled.

## How the test suite reports these

`test_rls_isolation.py` marks every assertion that depends on a known broken
table with `xfail(strict=False)`. That keeps the suite green while preserving
a paper trail: if any of these failures becomes a pass (RLS gets fixed), the
test moves from XFAIL → XPASS and surfaces in the summary.

To re-run after fixes land:
```
cd /root/chess-app/backend && \
  SUPABASE_DB_URL='postgresql://postgres:<pw>@db.<ref>.supabase.co:5432/postgres' \
  venv/bin/python -m pytest tests/test_rls_isolation.py -v
```
